The following is a detailed technical explanation of how a modern transformer language model performs autoregressive decoding on a many-core CPU, written for an audience of systems programmers who already understand linear algebra and cache hierarchies but want to learn the specific engineering tradeoffs involved.

When a transformer generates text one token at a time, each step is dominated by matrix-vector products rather than the large matrix-matrix products that characterize the prefill phase. During prefill, the model processes the entire prompt at once, so the activation tensor has many rows and the hardware can amortize the cost of streaming each weight matrix from memory across all of those rows; the computation is compute bound and the floating point units stay busy. During decode, by contrast, there is exactly one new token, so the activation is a single vector, every weight matrix is read from memory exactly once and multiplied by that one vector, and the computation becomes memory bandwidth bound. This asymmetry is the single most important fact to internalize before optimizing an inference engine, because it means the techniques that help prefill often do nothing for decode, and occasionally even hurt it.

Consider the attention mechanism specifically. In a standard dense attention layer the cost of decoding grows linearly with the context length, because the new query must be compared against every cached key. For very long contexts this term eventually dominates everything else, so production systems introduce sparse or compressed attention variants that select only the most relevant past positions. A lightweight indexer computes a cheap relevance score for every compressed position, then a top-k selection keeps only the highest scoring ones, and the expensive weighted value aggregation runs over that small selected set instead of the full history. The indexer scan itself is linear in context and reads activations rather than weights, so quantizing the model weights does nothing to accelerate it; the only way to speed it up is to make the scan arithmetic cheaper or to move fewer bytes per position. There are three complementary techniques that production inference engines combine to achieve this, and we will now examine each of them in careful detail. The first technique concerns the arithmetic itself. On a wide vector machine the score is a dot product between a query head and a cached key, and the natural implementation accumulates partial sums in single precision floating point. However, because both operands have already been block quantized at write time, the values they represent occupy only a small number of distinct levels, and a narrower integer representation loses essentially nothing. By storing the cached keys as eight bit integers and using the hardware dot product instruction that multiplies four integer pairs per lane, the engine performs the same number of multiply accumulate operations in a quarter of the instructions and reads a quarter of the bytes from memory.
