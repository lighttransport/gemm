# M3 — MiniMax-M3 (text) inference on A64FX/Fugaku

MiniMax-M3 (`~/models/m3` on the Fugaku frontend, **869 GB bf16**, 59 safetensors
shards, 23416 tensors) is a 428B-param (~23B activated) sparse-MoE model: 60 layers,
hidden 6144, GQA attention, MiniMax Sparse Attention (MSA), sigmoid-bias MoE, 1M
context. We run the **text LM only** on **96 A64FX nodes**, reusing the DS4P custom
C/SVE expert-parallel runner stack rather than llama.cpp/vLLM.

This is a **new forward graph**, not a DS4P config switch — M3's attention (GQA+MSA),
router (sigmoid+bias), activation (SwiGLU-OAI), RoPE (partial), and norm (Gemma) all
differ from DeepSeek-V4. We reuse the *infrastructure* (EP/TP sharding, the parallel
stager, the uTofu EP all-reduce lifted to N≤128, the bf16 SVE matvec, the pjsub job
patterns) and the *MSA SVE kernels* (`~/work/clair/main/tests/a64fx/msa`, which are
exactly M3's sparse-attention primitive — block 128, head_dim 128, top-K blocks).

Config + per-rank structs: **`common/m3.h`** (verified, cross-compiles with fccpx).
Forward graph + arena + loaders: **`common/m3_impl.h`** (TODO, Phase 2/3). Runner:
**`a64fx/m3/m3_ep_runner.c`** (TODO; near-copy of `a64fx/llm/ds4f_ep_runner.c`).

## Memory budget (96 nodes, bf16, text-only) — TP is MANDATORY

Per A64FX node = 32 GB. EP shards the 128 experts; **TP shards the replicated dense**.

- **Experts** (the bulk): per expert/layer = 3×(3072×6144) bf16 ≈ 113 MB; 128 experts ×
  57 MoE layers ≈ **826 GB**. EP over 96 ranks (128 experts → 32 ranks own 2, 64 own 1)
  ⇒ **≤ 12.9 GB/rank**.
- **Replicated dense if NOT TP'd** ≈ 25 GB/rank: embed+head ~5 GB, attn q/o ~12 GB
  (57×~210 MB), shared expert ~6.4 GB, dense-layer FFN ~1.4 GB. **12.9 + 25 ≈ 38 GB >
  32 GB ⇒ overflow.** So TP is not optional.
- **TP set** (same as DS4P): vocab-shard embed + head; head-shard q_proj/o_proj
  (TP_ATTN — kv heads + MSA indexer stay replicated, they're small); intermediate-shard
  shared expert (w1/w3 rows, w2 cols→EP-sum); intermediate-shard the dense FFN. With TP
  across the 96 ranks the replicated remainder drops several-fold ⇒ **target ≈ 18–22
  GB/rank**, comparable to DS4P's validated 22.99 GB/rank @96n. Validate fit via
  **MemFree, not RSS** (THP caches are RSS-invisible — this bit DS4P repeatedly).
- 128 experts / 96 ranks is **uneven** (2/1); the EP map `e % ep_size` handles it
  (DS4P ran 35/34 on 11 ranks). 96 nodes per the request; even-EP alternatives if ever
  wanted: 64 nodes (2 each) or 128 (1 each).

Shard helpers live in `common/m3.h`: `m3_shard` (even split + remainder),
`m3_shard_heads` (whole-head shard), `m3_n_owned` (EP expert count).

## Build + run workflow (cross-compile + pjsub — NOT native)

This environment is an **x86 mirror**; `ssh fugaku` is an x86 **login node** with the
**cross compilers** `fccpx`/`FCCpx` (no native `fcc`). The repo lives on the Fugaku
global FS at `/vol0006/mdt0/data/hp250467/work/gemm/ds4p/` (Mutagen-mirrored from the
local tree — local edits propagate). Build on the login node, run on compute nodes via
`pjsub`. (The CLAUDE.md "native A64FX, run directly with fcc" note does not apply here.)

```sh
# cross-compile (login node)
fccpx -Nclang -O3 -march=armv8.2-a+sve -ffp-contract=fast -I <repo> ...
# run: pjsub a job script (clone a64fx/ds4p/pjsub_ds4p_96n.sh -> pjsub_m3_96n.sh)

      --mpi proc=96 --llio localtmp-size=...Gi pjsub_m3_96n.sh
```

Staging + run share ONE pjsub job (`/local` is wiped per job). `PJM_LLIO_GFSCACHE=/vol0006`.

- **PRACTICAL CEILING (dummy compute, real comm+mem) — job 49242910, 96n, full 60L,
  TP on, 12 threads, synthetic.** `M3_DUMMY=1`: matvecs stream weight bytes at full HBM BW
  (no FMA), EP all-reduce + dispatch stay real. Result:
  - actual  (M=1 bf16 matvec): prefill **3.11** / decode **3.11 tok/s**, comm ~25%
  - ceiling (dummy stream):     prefill **3.28** / decode **3.25 tok/s**, comm ~23%
  **The ceiling is only ~5% above actual ⇒ compute is NOT the bottleneck.** Removing the
  M=1 FMA barely helps — the single-stream decode is bound by **dispatch (hundreds of
  matvec parallel-regions/token) + comm (~23%) + the serial non-matvec work** (attention,
  norms, router, SwiGLU), NOT by compute or raw bandwidth. So **the practical ceiling for
  this engine is ~3.3 tok/s**, vs the ~1,700 tok/s hardware BW ceiling — the ~500× gap is
  STRUCTURAL (single-stream, token-at-a-time M=1, per-matvec dispatch, exposed comm), not a
  kernel-efficiency problem. Real levers: batch/multi-stream (raise M), comm overlap, and
  fewer/bigger dispatches (megakernel).
  - **Node scaling (64 vs 96, job 49242917):** 64n actual 3.10 / ceiling 3.23 tok/s, comm
    ~31%; 96n actual 3.11 / ceiling 3.25, comm ~25%. **tok/s is ~flat in node count** — the
    single-stream per-rank work + comm dominate either way, so shrinking nodes neither helps
    nor hurts throughput; node count is chosen for memory/context capacity, not decode speed.
  - **1M context:** the GQA KV is **~120 GB/rank at 1M** (4 kv heads ×128 ×2B ×2(K+V) ×60L
    = 120 KB/pos), replicated ⇒ exceeds 32 GB at ANY node count without **context-parallel
    KV sharding or int4-KV** (not yet implemented). The ceiling above is ctx-independent
    (weights+comm+dispatch); 1M only adds the MSA index-scan read (~14.6 GB/rank, the only
    ctx-growing term) ⇒ estimated ~2.6–3.0 tok/s at 1M once CP/int4 lets it fit.
- **MULTI-STREAM BATCHED DECODE — ~4x aggregate throughput (jobs 49248886/49248983, 96n).**
  `m3_forward_batch_decode` runs N concurrent streams/forward: dense projections become
  batched GEMMs M=N (`m3_gemm_bf16`, K-tiled), the EP all-reduce fires ONCE per layer over
  the whole [N,hidden] partial, and the per-stream glue (RMSNorm/RoPE/attention/F32 router)
  is OpenMP-parallel across streams. Per-stream KV caches; routed experts M=1 per (token,
  expert) for v1. Synthetic full-60L, TP on, 12 threads:
  | N | agg tok/s | vs N=1 | comm% |
  |---|---|---|---|
  | 1 | 3.13 | 1.0x | 25% |
  | 4 | 8.82 | 2.82x | 37% |
  | 8 | 12.33 | 3.94x | 42% |
  The **structural lever works**: batching amortizes the dispatch+comm the dummy ceiling
  showed dominate (v1 glue-serial = +22% only; v2 glue-parallel = **~4x**). Bottleneck now
  shifts to comm (25%->42% as compute shrank) -> **comm-overlap is the next lever**.
  Runner `M3_MSTREAM=N` (`pjsub_m3_mstream_96n.sh`); aggregate is the serving-throughput
  metric. (1M ctx with N streams still needs CP/int4-KV; expert grouping batches the M=1
  routed experts further.)
